以下是对 **GitHub Copilot Chat v0.39.0** 扩展中 Claude 和 GPT 模型请求分流规则的逆向分析报告。

---

## 一、整体架构：三条 API 通道

扩展内部定义了一个请求类型枚举 `Zn`，包含三个核心聊天通道：

| 通道标识            | 后端路径                           | 协议格式                         |
| ------------------- | ---------------------------------- | -------------------------------- |
| `ChatCompletions` | `{capiBaseUrl}/chat/completions` | OpenAI Chat Completions          |
| `ChatResponses`   | `{capiBaseUrl}/responses`        | OpenAI Responses API             |
| `ChatMessages`    | `{capiBaseUrl}/v1/messages`      | **Anthropic Messages API** |

默认 `capiBaseUrl` 为 `https://api.githubcopilot.com`。所有请求统一走 GitHub Copilot 代理网关，由网关根据模型路由到实际的 OpenAI/Anthropic 后端。

## 二、模型族识别函数

扩展通过 `family` 字段（从 CAPI models 接口返回的元数据）判断模型厂商：

```javascript
// Claude 系列：family 以 "claude" 或 "Anthropic" 开头
function Ly(n) {
  return n.family.startsWith("claude") || n.family.startsWith("Anthropic") || doi(n);
}

// GPT 系列
function Vpn(n) {
  return n.family.startsWith("gpt") && !n.family.includes("gpt-4o") 
    || n.family === "o4-mini" || oFe(n.family) || ...;
}

// Gemini 系列
function vK(n) {
  return n.family.toLowerCase().includes("gemini") || NQ(n);
}
```

## 三、API 通道选择：核心分流逻辑

endpoint 对象的 `apiType` 属性决定最终走哪条通道，优先级从高到低：

```javascript
get apiType() {
  return this.useResponsesApi   ? "responses"       // → ChatResponses
       : this.useMessagesApi    ? "messages"         // → ChatMessages (Anthropic)
       :                          "chatCompletions"; // → ChatCompletions (OpenAI)
}
```

### 3.1 `useResponsesApi` 判定

```javascript
get useResponsesApi() {
  // 条件1: 元数据不支持 /chat/completions 但支持 /responses → 强制 responses
  if (supported_endpoints && 
      !supported_endpoints.includes("/chat/completions") && 
      supported_endpoints.includes("/responses")) return true;
  // 条件2: 只要支持 /responses 就可以用
  return !!supported_endpoints?.includes("/responses");
}
```

### 3.2 `useMessagesApi` 判定（Claude 专用）

```javascript
get useMessagesApi() {
  return !!(
    configService.getExperimentBasedConfig("chat.anthropic.useMessagesApi") &&
    modelMetadata.supported_endpoints?.includes("/v1/messages")
  );
}
```

**关键**：需要**同时满足**两个条件：

1. 实验标志 `chat.anthropic.useMessagesApi` 开启（默认 `true`）
2. 模型元数据的 `supported_endpoints` 包含 `/v1/messages`

### 3.3 分流总结

| 模型                       | `supported_endpoints`（服务端返回）                             | 实际走的通道                             |
| -------------------------- | ----------------------------------------------------------------- | ---------------------------------------- |
| GPT-4o / GPT-4.1 / o4-mini | `["/chat/completions", "/responses"]`                           | **ChatResponses** (responses 优先) |
| Claude Sonnet 4 / Opus 4   | `["/v1/messages"]` 或 `["/chat/completions", "/v1/messages"]` | **ChatMessages** (Anthropic 原生)  |
| Claude (旧版/fallback)     | `["/chat/completions"]`                                         | **ChatCompletions**                |
| Gemini                     | `["/chat/completions", "/responses"]`                           | **ChatResponses**                  |

## 四、响应处理分流

```javascript
async processResponseFromChatEndpoint(e, t, r, a, o, s, c) {
  return this.useResponsesApi  ? Yyn(...)    // Responses API 流式解析
       : this.useMessagesApi   ? zyn(...)    // Anthropic Messages 流式解析 (SSE)
       : this._supportsStreaming ? Xyn(...)  // OpenAI ChatCompletions SSE
       :                          gAi(...); // 非流式回退
}
```

Claude 的响应通过专门的 `u7` 类处理，该类解析 Anthropic 原生的 SSE 事件格式（`content_block_start`, `content_block_delta`, `thinking` 等）。

## 六、System Prompt 按模型定制

扩展通过 `Bc.registerPrompt()` 注册了按 `familyPrefixes` 分流的 prompt 解析器：

| familyPrefixes                              | Prompt 解析器 | System Prompt 特点                                                 |
| ------------------------------------------- | ------------- | ------------------------------------------------------------------ |
| `["claude", "Anthropic"]`                 | `OIt`       | 分三档：Sonnet 4 →`QIt`；Claude 4.5 → `W_e`；其他 → `Z_e` |
| `["gpt", "o4-mini", "o3-mini", "OpenAI"]` | GPT 解析器    | 统一 `KIt`                                                       |
| `["gemini"]`                              | Gemini 解析器 | 可选特殊 prompt `$It`                                            |
| CodeX (`gpt-5.2-codex`)                   | 专用解析器    | `ekt`                                                            |
| `vscModelA / vscModelB`                   | 内部实验模型  | 各自独立                                                           |

## 八、BYOK（Bring Your Own Key）分流

当用户配置自有 API Key 时，扩展内嵌了完整的 Anthropic SDK（`rp` / `Anthropic` 类），直接调用 `https://api.anthropic.com/v1/messages`，绕过 GitHub 网关。BYOK Anthropic 提供者名为 `"AnthropicBYOK"`，telemetry source 为 `"byok.anthropic"`。

---

## 九、Claude Agent（Claude Code 集成）

扩展内嵌了 Claude Code SDK（`claude-agent-sdk`），通过本地 HTTP 代理服务器 `ClaudeLanguageModelServer` 将 Claude Code 的 Anthropic Messages API 请求转发给 Copilot 网关。关键流程：

1. `ClaudeAgentManager` 创建 `ClaudeCodeSession`
2. Session 通过环境变量 `ANTHROPIC_BASE_URL=http://localhost:{port}` 将 Claude Code CLI 的请求劫持到本地代理
3. 代理服务器接收 `/v1/messages` 请求，通过 `handleAuthedMessagesRequest` 转发到 Copilot 的 `ChatMessages` 通道
4. 模型选择：endpoint fallback 逻辑将 `claude-haiku-4*` → `claude-haiku-4.5`，`claude-sonnet-4*` → `claude-sonnet-4.5`，`claude-3-5-haiku` → `gpt-4o-mini`（降级）

## 总结流程图

```
用户请求 → getChatEndpoint(model)
              ↓
     模型元数据（含 supported_endpoints）
              ↓
     ┌─── useResponsesApi? ──── Yes → ChatResponses → /responses  (GPT/Gemini)
     │         No
     ├─── useMessagesApi?  ──── Yes → ChatMessages  → /v1/messages (Claude)
     │         No
     └──────────────────────────────→ ChatCompletions → /chat/completions (fallback)
              ↓
     统一经 GitHub Copilot 网关 (api.githubcopilot.com) 转发到后端
```

**核心判据**：分流不是在客户端硬编码模型名决定的，而是由**服务端通过 models API 返回的 `supported_endpoints` 元数据**驱动。客户端仅根据这些端点声明 + 实验标志来选择 API 协议格式。**核心判据**：分流不是在客户端硬编码模型名决定的，而是由**服务端通过 models API 返回的 `supported_endpoints` 元数据**驱动。客户端仅根据这些端点声明 + 实验标志来选择 API 协议格式。
